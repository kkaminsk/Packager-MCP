# Technical Design: Intune Setup Script

## Context

The `publish_to_intune` tool requires certificate-based service principal authentication. Currently, users must manually:
1. Navigate Azure portal to create app registration
2. Generate certificates using PowerShell or OpenSSL
3. Upload certificates to Azure
4. Configure API permissions
5. Grant admin consent
6. Set environment variables

This design introduces an automated setup script based on proven patterns from the reference function library.

### Stakeholders
- IT Admins setting up packager-mcp for first time
- MSPs deploying to multiple client tenants
- DevOps engineers automating MCP server deployment

## Goals / Non-Goals

### Goals
- One-command setup experience for Intune integration
- Reusable PowerShell function library following established patterns
- YAML configuration file as alternative to environment variables
- Idempotent execution (safe to re-run)

### Non-Goals
- Azure Key Vault integration (future enhancement)
- Multi-tenant management from single config
- Automated certificate renewal (manual process)
- Windows Credential Manager integration (future enhancement)

## Decisions

### Decision 1: PowerShell 7+ with Microsoft.Graph Module

**What**: Use PowerShell 7+ and Microsoft.Graph PowerShell SDK for all Azure operations.

**Why**:
- Cross-platform (Windows, macOS, Linux)
- Microsoft.Graph is the official, supported SDK
- Consistent with reference function patterns
- Better async support than Windows PowerShell 5.1

**Trade-offs**:
- Requires PowerShell 7 installation (not default on Windows)
- Microsoft.Graph module is large (~100MB)

### Decision 2: YAML Configuration File

**What**: Store configuration in `intune_mcp_config.yaml` rather than JSON or environment variables alone.

**Why**:
- YAML is more readable for humans
- Consistent with existing config patterns (e.g., Docker Compose)
- Supports comments for documentation
- Already have `yaml` package in project dependencies

**Schema**:
```yaml
# Packager-MCP Intune Configuration
# Generated by Setup-PackagerMcpIntune.ps1

intune:
  # Azure Entra ID tenant
  tenant_id: "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

  # App registration client ID
  client_id: "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"

  # Path to certificate file (PFX or PEM)
  certificate_path: "./certs/packager-mcp.pfx"

  # Certificate thumbprint (for verification)
  certificate_thumbprint: "AABBCCDD..."

  # Certificate expiration (for monitoring)
  certificate_expires: "2026-12-27T00:00:00Z"

# Metadata
_generated:
  timestamp: "2025-12-27T12:00:00Z"
  script_version: "1.0.0"
  generator: "Setup-PackagerMcpIntune.ps1"
```

### Decision 3: Function Library Structure

**What**: Modular PowerShell functions in `scripts/functions/` directory.

**Why**:
- Matches reference function library pattern
- Each function is testable independently
- Easy to maintain and extend
- Companion `.md` documentation files

**Structure**:
```
scripts/
├── Setup-PackagerMcpIntune.ps1    # Main orchestration script
├── functions/
│   ├── Write-PackagerMcpLog.ps1
│   ├── Write-PackagerMcpLog.md
│   ├── Invoke-GraphWithRetry.ps1
│   ├── Invoke-GraphWithRetry.md
│   ├── Connect-PackagerMcpGraph.ps1
│   ├── ... (other functions)
│   └── claude.md                   # Function library documentation
└── README.md                       # Setup instructions
```

### Decision 4: Self-Signed Certificate Generation

**What**: Generate self-signed certificates by default, with option to use existing certificates.

**Why**:
- Simplest path for initial setup
- No external dependencies (CA, Key Vault)
- 2-year validity is sufficient for most use cases
- Users can replace with CA-signed certs later

**Implementation**:
```powershell
$cert = New-SelfSignedCertificate `
  -Subject "CN=Packager-MCP" `
  -CertStoreLocation "Cert:\CurrentUser\My" `
  -KeyExportPolicy Exportable `
  -KeySpec Signature `
  -KeyLength 2048 `
  -KeyAlgorithm RSA `
  -HashAlgorithm SHA256 `
  -NotAfter (Get-Date).AddYears(2)
```

### Decision 5: Config Precedence

**What**: Environment variables take precedence over YAML config.

**Why**:
- Allows runtime overrides without modifying files
- Consistent with 12-factor app principles
- Docker/container deployments can use env vars
- Local development can use YAML

**Precedence order**:
1. Environment variables (`AZURE_TENANT_ID`, etc.)
2. YAML config file (`intune_mcp_config.yaml`)
3. Default values (none - fail if not configured)

## API Design

### Setup Script Parameters

```powershell
Setup-PackagerMcpIntune.ps1
  [-DisplayName <string>]        # App registration name (default: "Packager-MCP")
  [-CertificatePath <string>]    # Path to existing PFX/PEM certificate
  [-CertificatePassword <SecureString>]  # Certificate password
  [-ConfigPath <string>]         # Output path for YAML config
  [-CertOutputPath <string>]     # Output path for generated certificate
  [-TenantId <string>]           # Target tenant (optional, uses default)
  [-Force]                       # Skip confirmation prompts
  [-TestConnection]              # Verify setup after completion
  [-Uninstall]                   # Remove app registration and cleanup
  [-Verbose]                     # Enable debug logging
```

### TypeScript Config Interface

```typescript
interface IntuneMcpConfig {
  intune: {
    tenant_id: string;
    client_id: string;
    certificate_path: string;
    certificate_thumbprint?: string;
    certificate_expires?: string;
  };
  _generated?: {
    timestamp: string;
    script_version: string;
    generator: string;
  };
}
```

## Risks / Trade-offs

### Risk 1: PowerShell 7 Requirement
**Risk**: Users may not have PowerShell 7 installed.
**Mitigation**: Script checks version and provides install instructions. Consider adding Windows PowerShell 5.1 compatibility.

### Risk 2: Microsoft.Graph Module Size
**Risk**: Large module download (~100MB) on first run.
**Mitigation**: Script checks for module and prompts before install. Document minimum required sub-modules.

### Risk 3: Certificate Password Storage
**Risk**: Users may store certificate passwords insecurely.
**Mitigation**:
- Don't store password in YAML by default
- Prompt at runtime
- Document Windows Credential Manager option for future

### Risk 4: Admin Consent Requires Elevated Privileges
**Risk**: User may not have Global Admin or Application Admin role.
**Mitigation**:
- Check permissions early and fail fast with clear message
- Document required roles in README

## Migration Plan

This is a new feature, no migration required.

### Rollout Steps
1. Release PowerShell function library
2. Release setup script with documentation
3. Update MCP server to read YAML config
4. Update CLAUDE.md with setup instructions
5. Consider adding setup prompt to MCP server startup
